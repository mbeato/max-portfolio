## Color

All values from globals.css @theme. The 60/30/10 role split below applies to visual weight, not pixel count.

| Role | Value | CSS Token | Usage |
|------|-------|-----------|-------|
| Dominant (60%) | #FAFAF9 | --color-map-white | Page background, section backgrounds — no alternating section colors |
| Secondary (30%) | #FFFFFF | (pure white) | Card surfaces (project cards, contact form card), input fields |
| Stone neutrals | #1A1A1A – #F5F5F4 | --color-stone-900 through --color-stone-100 | Text hierarchy, borders (via shadow), tags, muted labels |
| Accent (10%) | #E8523F | --color-coral-peak | Exact reserved list — see below |
| Accent hover | #C7402F | --color-coral-deep | Pressed/active state of coral elements only |
| Destructive | #DC2626 | --color-error | Form validation error messages only — not the accent coral |
| Success | #16A34A | --color-success | Form submission success state only |

Accent (#E8523F) reserved for exactly these elements in Phase 5:
1. Active nav link state (underline + color shift)
2. Resume button in nav (primary button background)
3. Experience card left border on hover (2px transition from #E5E5E5)
4. Project card left border on hover (2px, appears on hover)
5. "view case study" CTA button on each project card (primary button)
6. "send message" submit button in contact form (primary button)
7. Contact form input focus ring (1px coral shadow)
8. "explore" scroll indicator hover state (color transition from stone-400)
9. Ghost link underline hover transition (stone-300 → coral)
10. Focus rings across all interactive elements (keyboard navigation)

Coral must NOT appear on: backgrounds, large surfaces, section dividers, icons at rest, body text, any decorative use.

Shadows (from globals.css :root, not @theme):
- `--shadow-border`: `rgba(0, 0, 0, 0.06) 0px 0px 0px 1px` — cards and containers at rest
- `--shadow-border-dark`: `rgba(0, 0, 0, 0.12) 0px 0px 0px 1px` — secondary button border
- `--shadow-hover`: `rgba(0, 0, 0, 0.04) 0px 1px 3px, rgba(0, 0, 0, 0.08) 0px 0px 0px 1px` — card hover
- `--shadow-lifted`: `rgba(0, 0, 0, 0.08) 0px 4px 12px -4px, rgba(0, 0, 0, 0.06) 0px 0px 0px 1px` — project cards
- `--shadow-focus`: `0 0 0 2px #FAFAF9, 0 0 0 4px rgba(232, 82, 63, 0.5)` — keyboard focus double-ring

Use boxShadow CSS property with these var() references — never CSS border for containment.

## Scroll Animation Treatment Matrix

Each section MUST use a distinct animation — SECT-02 requirement. Uniform fade-in-on-scroll is explicitly forbidden (SLOP-05).

| Section | Variant | Trigger | Notes |
|---------|---------|---------|-------|
| Hero subtitle | `fadeIn` | puzzleSolved after 1.8s delay | No transform on active canvas |
| TopoSvgDivider | `drawPath` per path | 30% viewport intersection | 0.15s stagger between paths |
| About — text | `slideInLeft` | 20% threshold | Elevation metaphor: content rises from left |
| About — photos | `fadeIn` | 20% threshold | No transform on image-heavy content |
| Experience list | `slideInLeft` + `staggerContainer` | 20% threshold | 0.08s stagger, entries enter sequentially |
| Projects list | `fadeInUp` + `staggerContainer` | 15% threshold | Cards rise upward, 0.08s stagger |
| Contact header | `fadeInUp` | 20% threshold | Standard reveal |
| Contact columns | `fadeInUp` + stagger (0.1s delay between columns) | 20% threshold | Left then right column |

All variants already defined in `src/lib/motion.ts`. Do not add new variants — use existing vocabulary.
Reduced-motion: all variants must use the `MotionProvider` context or check `useReducedMotion`. The `fadeIn` variant (opacity only) is the safe fallback for all reduced-motion users.
